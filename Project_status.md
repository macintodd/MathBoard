Each ink tool remembers its own palette preset and editable five-color strip. Pen defaults to a 10-point stroke width. In both the quick strip and full drawer, tapping a swatch selects it and tapping the same swatch again opens the color picker for that swatch.

Extract is reserved for region/fragment selection and owns the Lasso/Box mode control. Extract now shows a no-text mini palette for Lasso/Box on first tap and opens the full submenu on second tap; Eraser similarly shows a no-text mini palette for Pixel/Stroke on first tap and opens the full submenu on second tap. Pixel eraser now maps to PencilKit's adjustable fixed-width bitmap eraser, with a 16-80 size slider and 24-point default; Stroke mode remains whole-stroke deletion where width is not meaningful. First floating Action HUD slice is implemented for selected text objects: Edit opens the full text editor, Clone duplicates, and Delete removes the object. The HUD now sits lower and closer to the selected text box. Canvas LaTeX rendering uses extra internal math padding/headroom and math-aware text-object fitting so superscripts and tall glyphs are less likely to clip or wrap poorly. The off-screen SwiftUIMath image render now uses fixed intrinsic sizing with generous SwiftUI padding instead of forcing Math into a tight frame; if simple monomials like `3x^6` still render incorrectly, evaluate replacing SwiftUIMath or bypassing off-screen image rendering for whiteboard math. Selected object drag hit testing also has a larger edge ownership margin so quick finger drags starting near the edge of a selected text/sticker object remain object drags. ## 1. One-paragraph summary

MathBoard is a teaching whiteboard for live high-school math tutoring (Algebra 2, Pre-Calculus, Geometry, AP Pre-Calculus) that ships for both **iPad and Mac**, with distinct roles per platform. The **iPad is the live-teaching surface** — Apple Pencil drawing, real-time presentation to a TV, and interaction with students during a session. The **Mac is the prep surface** — building presentations before class by importing PDFs, photos, and GIFs (Graphics Interchange Format animated images), arranging slides, adding elements, and doing crude mouse-based annotation; live-teaching parity on Mac is not a goal. Students join sessions from a browser (no install required) via the screen-sharing companion site that will be built in a later phase. The session host caps at 5 participants including the teacher. Core technical pillars: SwiftUI for the interface, PencilKit for drawing, PDFKit for handouts, LiveKit for real-time voice and drawing sync, iCloud Drive `.mathboard` file packages for storage.

## 2. Platform and compatibility decisions

- **Platforms (confirmed):** **iPad and Mac.** No iPhone build, no visionOS — at least not for v1.
- **Per-platform roles (confirmed):**
  - **iPad — live teaching surface.** Full feature set: Apple Pencil drawing, slide navigation, external display presentation, freeze/unfreeze, host-side session controls, real-time student collaboration.
  - **Mac — teacher prep surface.** Used **before class** to assemble presentations: import PDFs, photos, and animated GIFs; arrange and reorder slides; add elements (text, shapes, images); annotate with crude **mouse-based** drawing. **Live-teaching parity on Mac is not a goal.** Apple Pencil support on Mac is explicitly not needed.
- **Cross-platform strategy (confirmed): native Multiplatform SwiftUI target.** One Xcode app target compiled for both iPadOS and macOS, with `#if os(iOS)` / `#if os(macOS)` guards only where a platform genuinely differs. Reasons:
  - Modern SwiftUI handles 80%+ of the layout work for both platforms with the same code.
  - Native (rather than Mac Catalyst) gives us real macOS look-and-feel and full access to AppKit-only APIs when needed.
  - One codebase = less to maintain for a solo novice developer.
- **Drawing on Mac — how it works:** Mac uses the same `.mathboard` file format and the same PencilKit `PKDrawing` data structure for stroke storage (so files round-trip iPad ↔ Mac perfectly), but the input is **mouse/trackpad only** and the Mac UI does **not** show the PencilKit floating tool picker. The Mac drawing experience is intentionally simple — a fixed toolbar with pen, eraser, and a small color/width selector. Anything fancier (pressure curves, tilt, barrel-roll) only matters on iPad.
- **Minimum OS versions (confirmed): iPadOS 26.0 and macOS 26.0.** Reasoning:
  - Modern-app baseline; both releases shipped fall 2025.
  - Unlocks **Liquid Glass**, Apple's new dynamic material design system introduced at WWDC 2025. Standard SwiftUI components automatically pick up Liquid Glass when built against the latest SDKs. This fits the "warm, playful, approachable" design language without us doing custom theming work.
  - Unlocks the latest PencilKit content version (`PKContentVersion.version3` — includes barrel-roll angle data from Apple Pencil Pro), giving us the best ink expressiveness on iPad without writing backward-compatibility fallbacks.
  - **Trade-off:** any iPad or Mac older than what iPadOS 26 / macOS 26 supports cannot run the app. Acceptable for v1 given the design goals.
  - **Acronym key:** API = Application Programming Interface, the set of functions Apple gives us to call. SDK = Software Development Kit, the bundle of headers and tools we build against. GIF = Graphics Interchange Format, an animated image format.
- **Companion website (future phase):** A separate web project — likely served from the same domain — for **screen-share-only** participation, modeled on how Explain Everything (explaineverything.app) lets guests watch in a browser. Students see the teacher's canvas and can draw on their own layer; no iPad or Mac required. **Not in scope for v1**, but the iPad and Mac apps' architecture must not block it.

## 3. Architecture (modular, future-proofed)

The app is structured so each capability lives in its own module. Modules are managed with **Swift Package Manager** (SPM — Apple's official tool for splitting code into reusable libraries). Concretely, we will create one local Swift package in the repo named `MathBoardCore` with one library target per module. This is simpler than separate packages for a solo developer and still keeps each module independently testable.

**Core modules (built first):**

| Module | Responsibility |
|---|---|
| `Documents` | Start screen, folder browser, `.mathboard` file package read/write, iCloud Drive integration, duplicate-set creation flow |
| `Canvas` | Layer stack (drawing, PDF, image, GIF, video, browser/streaming, HTML applets via WKWebView, shapes, calculator/graph), each participant on their own named layer. **iPad**: PencilKit canvas with full Apple Pencil support, floating tool picker. **Mac**: same `PKDrawing` storage format for file compatibility, but a simpler mouse-driven canvas with a fixed toolbar — no floating tool picker, no Pencil-specific input handling. |
| `Slides` | Multi-page support, slide navigator pill, filmstrip popover |
| `Presentation` | External display via HDMI or AirPlay, mirror/present toggle, 16:9 viewfinder over the 4:3 iPad canvas (TV crops left/right), Move Window vs. Move Canvas viewport modes, freeze/unfreeze, optional Apple Pencil pointer |
| `Collaboration` | LiveKit voice + drawing sync, session codes, host controls (mute, clear, freeze student layers), 5-participant cap |

**Feature modules (added after core is working):**

| Module | Responsibility |
|---|---|
| `CalculatorGraphing` | Scope: Algebra 2 Big Ideas Math Chapters 1–7; Desmos-style sliders |
| `StudentWebViewer` | Browser-based screen-share companion (separate web project; lives in its own repo or sub-folder later) |
| `Quizzes` | Light formative-assessment tools |
| `Analytics` | Teacher-facing usage and engagement metrics |
| `MathManipulatives` | Interactive geometric / algebraic widgets |

**Acronym key:**
- **SwiftUI:** Apple's modern declarative user interface framework.
- **PencilKit:** Apple's framework for Apple-Pencil drawing.
- **PDFKit:** Apple's framework for displaying and annotating PDFs.
- **WKWebView:** Apple's class for embedding a web page inside a native app.
- **LiveKit:** Third-party real-time audio/video/data platform; we use it for voice and stroke synchronization.
- **HDMI:** High-Definition Multimedia Interface — the standard cable that connects an iPad (via adapter) to a TV.
- **AirPlay:** Apple's wireless screen-mirroring protocol.

- **`Project_status.md`** at the repo root (this file).
- **`MathBoardCore` Swift package** at the repo root.
  - `MathBoardCore/Package.swift` — declares library products including `Documents` and `ToolPalette`. Main app dependency chain is `Documents` → `Slides` → `Presentation` → `Canvas`; `Presentation` also depends on `ToolPalette` for the default-OFF visual radial palette overlay. Platforms: iOS 26 and macOS 26. `swift-tools-version: 6.2`.

Slides, Presentation, and Canvas are transitively linked through Documents — no separate Xcode framework-link step is needed when adding new internal targets to the package.
- **Documents module (inside `MathBoardCore/Sources/Documents/`)** — seven files:
  - `DocumentModels.swift` — `Folder` and `Lesson` value types backed by real filesystem URLs (no sample data). `DocumentMetadata` is the `Codable` representation of `document.json`. All internal.
  - `DocumentStore.swift` — **public** `@MainActor @Observable` class. Owns Documents-directory I/O. Loads folders + recent lessons, lists lessons in a folder, creates/renames/deletes folders, creates new lessons, renames lessons, deletes lessons singly or in batches, duplicates lessons singly or in batches of up to 10, moves lessons between folders with conflict-safe names, imports outside `.mathboard` packages into Imported Lessons, opens already-managed packages without duplicating them, and bootstraps starter content on first launch. Only `DocumentStore` itself and `init()` are public; the rest of the API stays internal to the module.
  - `AppColors.swift` — warm/playful color tokens (canvas cream, folder mustard, terracotta accent). Internal.
  - `NameEntrySheet.swift` — internal reusable modal sheet. Takes a title, placeholder, confirm-label, and `(String) -> Void` callback. Drives the New Folder and New Lesson flows.
  - `StartScreenView.swift` — **public** entry point. Reads `store.folders` and `store.recentLessons` via `@Environment(DocumentStore.self)`. Toolbar actions are **Open Lesson** (imports outside `.mathboard` packages or opens already-managed packages) and **New Folder** (presents `NameEntrySheet`, calls `store.createFolder`). Folder tiles open on tap and expose Rename/Delete from the context menu; folder delete requires confirmation because it removes all lessons inside. Recent lesson rows open directly. Empty state appears when no folders exist.
  - `FolderDetailView.swift` — internal. Loads `store.lessons(in: folder)` inside a `.task { }` modifier on appear. Toolbar action is **New Lesson** (presents `NameEntrySheet`, calls `store.createLesson`, refreshes the local lessons list). Lesson rows open on tap and expose Rename, Duplicate, and Delete from the context menu; Duplicate opens a compact count overlay capped at 10 copies, and Delete requires confirmation. The toolbar includes Select/Done for multi-select; selected lessons can be moved to another folder or bulk deleted with confirmation. Empty state appears for an empty folder.
  - `LessonDetailView.swift` — internal. Thin wrapper that embeds `SlidesView(lessonURL: lesson.url)` (from Slides module). Sets `DisplayBroker.shared.lessonURL` on appear so the broker tracks the current lesson across slide navigation. On iPad it **hides the system navigation bar** (`.toolbar(.hidden, for: .navigationBar)`) so the whiteboard can span the full screen, and instead renders a floating `lessonChrome` overlay in the top-leading corner: a circular Back button plus the lesson title in a material capsule. Only the Back button is interactive; the title uses `.allowsHitTesting(false)` so Apple Pencil strokes pass through to the canvas beneath it. The overlay sits inside the safe area (below the status bar).
- **Presentation module (inside `MathBoardCore/Sources/Presentation/`)** — four files:
  - `PresentingCanvasView.swift` — **public** wrapper around `CanvasView`. `init(drawingURL: URL)` — takes a slide-specific drawing file URL, not a lesson URL, plus optional `onImportPDF` / `onExportPDF` closures supplied by Slides. Composes the drawing surface with the toggleable viewfinder in a `canvasStack` that **ignores all safe areas** so the whiteboard (and the 16:9 viewfinder) reaches every edge of the display. Passes a **static-method** reference (`Self.publishFrame`) as the `onFrameUpdate` callback to `CanvasView` — stable identity across body re-renders, prevents SwiftUI AttributeGraph cycles. The former navigation-bar toolbar is gone; all of its controls now live in a floating `floatingToolMenu` — a single top-trailing `•••` `Menu` (safe-area-respecting) containing Undo/Redo, Zoom In/Out/Fit/Reset (with the live zoom % as the section header), the Present/Mirror toggle, Calculator, Graph Calculator, the Tool Palette submenu (show/hide, style, size), and Import/Export PDF. A green `tv.fill` badge appears next to the `•••` button when an external display is connected. Also hosts the default-OFF `FloatingToolPaletteView` overlay when `ToolPaletteSettings.shared.isCustomPaletteEnabled` is true and translates Pen/Marker/Eraser palette commands into Canvas tool commands.
  - `ViewfinderOverlay.swift` — internal. Fixed 16:9 rectangle with terracotta border, centered, non-interactive. Outside is dimmed at 35% black via even-odd `Path` fill rule.
  - `DisplayBroker.swift` — **public** `@MainActor @Observable` singleton bridging the iPad scene and any external-display scene. Holds `lessonURL`, `currentFrame: CGImage?`, `isExternalDisplayConnected`, calculator reference sizing, and the visual radial-palette state/position used by external display mirroring. `CGImage` because it's cross-platform (Presentation builds for iOS and macOS) and avoids a per-frame PNG/JPEG encode-decode round trip. Singleton because UIScenes do not share SwiftUI environment values.
  - `ExternalCanvasView.swift` — **public** SwiftUI view shown on a connected external display. Observes `DisplayBroker.shared.currentFrame`; uses `GeometryReader` + an explicit-by-hand `.frame(width:height:)` fitted-size computation (instead of `.aspectRatio(.fit)`) to scale the image to the maximum size that preserves the source aspect ratio within the TV's bounds. Larger graduation-cap "MathBoard" splash (144pt icon, 96pt title) when no lesson is loaded yet. Calculator overlay is visible on the external display in both modes. The tool palette is visible on the external display only in full iPad sharing (Mirror mode) and is hidden in Present Mode. To keep the mirrored palette aligned and in sync, the external display reuses the **same** `FloatingCompactToolPaletteView` the iPad uses — laid out in the iPad reference-size space and then uniformly scaled onto the fitted TV rect (so the rail/drawer dock offset can't drift). The compact drawer-vs-mini-strip choice mirrors via `ToolPaletteState.isCompactDrawerOpen` (shared state), rather than local view state that the TV instance couldn't see.
  - `ToolPaletteSettings.swift` — **public** `@MainActor @Observable` feature flag for the custom radial palette, persisted in `UserDefaults`, default OFF.
- **Slides module (inside `MathBoardCore/Sources/Slides/`)** — six files:
  - `SlideMetadata.swift` — `Codable`, `Identifiable`, `Sendable` per-slide metadata (`id: UUID`, `createdAt: Date`) plus optional persisted viewport state (`zoomScale`, `contentOffsetX`, `contentOffsetY`) and optional PDF page background reference.
  - `SlideStore.swift` — **public** `@MainActor @Observable` class. Owns the slide list for a single `.mathboard` package. Loads / writes `<lesson>.mathboard/slides.json`. Supports add, delete, manifest-order moves, per-slide viewport updates, and importing selected PDF pages as separate slides; delete removes the slide's `.drawing` file and prevents deleting the final remaining slide. PDF import can reuse the current slide when it is blank, otherwise it inserts immediately after the current slide. **Migration** on first open of a v1 single-page document: synthesizes one `SlideMetadata`, renames `strokes/main.drawing` → `strokes/slide-<uuid>.drawing`, writes a `slides.json`. After migration, the lesson behaves like any other multi-slide lesson.

Restores each slide's saved zoom/pan viewport when opened and debounces viewport writes back to `slides.json`. Import/Export PDF are no longer nav-bar toolbar items; SlidesView passes `onImportPDF` / `onExportPDF` closures down to `PresentingCanvasView`'s floating `•••` menu. Import uses SwiftUI `fileImporter`, copies the selected PDF to a temporary preview file, shows selectable page thumbnails, imports selected pages into `assets/`, and jumps to the first imported page slide. Handles delete confirmation and slide-operation error alerts.

Thumbnail grid for PDF import; all pages selected by default, tap thumbnails to select/deselect, Select All / Select None actions, and Import disabled until at least one page is selected. Body dispatches to the platform-specific drawing surface via `#if os(iOS)` / `#if os(macOS)`. No dependency on Documents (so Documents → Canvas is one-directional).

Owns live-stroke path shaping for the external-display overlay so smoothing, pressure/taper behavior, and future custom-rendered ink can evolve in one Canvas file without changing PencilKit storage. Current renderer filters near-duplicate samples, uses cubic Bezier smoothing, emits variable-width/tapered translucent marker segments, and uses a crisper single-path stroke for opaque pen ink.
  - `CanvasTextObject.swift` — public text-object model and JSON sidecar helpers. Each slide can store text in `strokes/slide-<uuid>.textobjects.json`; Mac edits the objects, while iPad/external display/PDF export render them read-only for now.
  - `CanvasGeometryObject.swift` — public vector geometry-object model (`CanvasGeometryShape` = line/circle/rightTriangle/triangle/rectangle/polygon, `CanvasGeometryArrow`) + JSON sidecar helpers. Frames are stored in canvas source coordinates (signed for lines so direction survives; `normalizedFrame` standardizes for hit-testing/rendering). Sidecar: `strokes/slide-<uuid>.geometryobjects.json`. The Canvas module has no ToolPalette dependency, so shape/arrow are Canvas-local enums that Presentation maps `GeometryType`/`GeometryLineArrowMode` onto.
  - `CanvasGeometryRenderer.swift` — public, cross-platform CoreGraphics drawing for geometry objects (fill + stroke + line arrowheads). Shared by the on-canvas overlay (`CanvasGeometryObjectsView` in `PencilKitCanvas.swift`), the external-display committed frame, and PDF export so all three stay identical. Callers pass already-mapped y-down target coordinates; the PDF exporter flips its y-up context to reuse it.
  - `PencilKitCanvas.swift` — iOS-only (whole file inside `#if os(iOS)`). `PencilKitCanvasContainer` owns the in-memory `PKDrawing` as `@State`, loads from disk in `.task`, saves on every `onChange`, renders PDF backgrounds and text objects with the ink, and publishes committed external-display frames. Live external ink uses the snapshot-free vector overlay path. It now consumes `CanvasToolCommand`: Pen/Marker become `PKInkingTool`, Eraser becomes `PKEraserTool`, Laser disables PencilKit's drawing recognizer and renders as a transient local/TV overlay, Text/equation disables PencilKit drawing and creates a persisted text object on tap with an active `UITextView` editor, Selection Object mode selects/moves/resizes/duplicates/deletes text objects, and Selection Region mode draws a visible lasso or marquee overlay as a first pass. The system `PKToolPicker` is hidden while the custom palette flag is enabled. The coordinator caches the active radial-palette Pen/Marker/Laser style so the TV live vector overlay uses the intended marker opacity/width and independent laser color/diameter.
  - `MacCanvasPlaceholder.swift` — macOS prep canvas. Loads/saves the same slide `.drawing` file as iPad, renders PDF page backgrounds as crisp vector-backed paper, renders/edits text objects, and turns mouse drags into compatible `PKStroke` values inside a `PKDrawing`. Includes simple undo, clear, zoom, pen/eraser modes, pen-width controls, and color presets.
- **App target (inside `MathBoard/MathBoard/`)** — five files plus a manual `Info.plist`:
  - `MathBoardApp.swift` — `@main` entry; constructs `DocumentStore()` as `@State` and injects it via `.environment(documentStore)`. Installs `AppDelegate` via `@UIApplicationDelegateAdaptor` (iOS-only via `#if canImport(UIKit)`).
  - `AppDelegate.swift` — UIKit `UIApplicationDelegate`. Fallback `configurationForConnecting` only — real scene-to-delegate routing happens via the Info.plist scene manifest.
  - `ExternalDisplaySceneDelegate.swift` — `UIWindowSceneDelegate`. On `willConnectTo`, creates a `UIWindow` hosting `ExternalCanvasView` and flips `DisplayBroker.shared.isExternalDisplayConnected = true`.
  - `ContentView.swift` — `import Documents` + hosts `StartScreenView` inside a `NavigationStack`.
  - `Info.plist` — custom file that declares `UIApplicationSceneManifest` with `UIWindowSceneSessionRoleExternalDisplayNonInteractive` mapped to `MathBoard.ExternalDisplaySceneDelegate`. Also enables Files app document visibility with `UIFileSharingEnabled = YES` and `LSSupportsOpeningDocumentsInPlace = YES`. Required: the project's `INFOPLIST_KEY_UIApplicationSceneManifest_Generation` build setting must be **NO** for both Debug/Release across all SDK overrides — otherwise the auto-generated scene manifest overwrites the custom one and external display routing breaks.
  - `Assets.xcassets`.
- **Navigation pattern in use:** SwiftUI value-based navigation, two levels deep. Both `Folder` and `Lesson` are `Hashable`. Tapping a folder tile pushes `Folder` (handled by `StartScreenView`'s `.navigationDestination(for: Folder.self)`); tapping a lesson row pushes `Lesson` (handled by `FolderDetailView`'s `.navigationDestination(for: Lesson.self)`). Each destination is registered close to where it is used. Works identically on iPad and Mac, survives state restoration cleanly.
- **Visibility convention inside the Documents module:** only the types the app target actually constructs are `public` (currently `StartScreenView` and `DocumentStore`). All other view, model, and helper types stay `internal`. Keeps the public surface minimal so future API breaks are unlikely.
- **`.mathboard` file package format (v2).** Each lesson is a directory ending in `.mathboard` containing:
  - `document.json` — lesson-level metadata: `{ id: UUID, createdAt: ISO 8601, version: 1 }`. Owned by Documents module.
  - `slides.json` — slide list, optional per-slide viewport state, and optional PDF page background reference: `{ slides: [{ id, createdAt, viewport, background }] }`. Owned by Slides module.
  - `strokes/slide-<uuid>.drawing` — one binary `PKDrawing` file per slide. Owned by Canvas module via Slides.
  - `strokes/slide-<uuid>.textobjects.json` — optional text-object sidecar for that slide. Owned by Canvas module.
  - `assets/` — stores imported PDF files copied into the lesson package.
  - Module-owned-file pattern: each module reads / writes only its own JSON, so they evolve independently. Documents live in the app's sandboxed Documents directory, are visible through Files app sharing, and can be imported through the app's Open Lesson flow.
- **Migration from v1 → v2 package format.** v1 packages had a single `strokes/main.drawing` and no slides list. On first open after upgrade, `SlideStore.loadOrMigrate` synthesizes one `SlideMetadata`, renames `main.drawing` → `slide-<uuid>.drawing`, and writes the new `slides.json`. After that the package is indistinguishable from a freshly-created multi-slide package.
- **First-launch bootstrap.** When `DocumentStore` finds an empty Documents directory it creates 5 starter folders (Algebra 2, Pre-Calculus, Geometry, AP Pre-Calculus, Tutoring Sessions) with 9 sample lessons total. After that the data is real and persists across launches.
- **Drawing persistence.** Each slide stores one binary `PKDrawing` file in `strokes/slide-<uuid>.drawing`. Saves are **debounced 400ms** — each stroke change cancels any pending save and schedules a fresh one; on view disappear (e.g., navigating back) any still-pending save flushes synchronously so work isn't lost. iPad writes through `PKCanvasView`; Mac writes compatible `PKDrawing` data from custom mouse-generated `PKStroke` values.
- **Real-time external display mirroring.** When an external display is connected, the iPad publishes committed `CGImage` frames to `DisplayBroker.shared.currentFrame` and a temporary vector live stroke to `DisplayBroker.shared.currentLiveStroke`.
  - **Committed renders** composite the visible PDF paper/background, PencilKit ink, and text objects into the active output rectangle.
  - **Live ink** comes from an Apple Pencil gesture recognizer that eavesdrops raw touch samples. For Pen/Marker it does not interrupt PencilKit; for Laser it disables PencilKit's drawing recognizer so the pointer/trail is presentation-only and is not saved into `PKDrawing`. Samples include location, pressure, and timestamp. The iPad publishes source-coordinate live strokes, and the external display projects active/recently completed vector strokes into the current viewport using `CanvasVectorInk`. Opaque Pen strokes use a crisp single-path renderer; translucent Marker/highlighter strokes use a full-width translucent smoothed path. Laser supports dot/trail live overlay rendering and clears after the palette fade duration. Normal pen/marker commits do not immediately force a TV raster refresh, so the TV does not visibly swap from MathBoard vector ink to PencilKit ink after each stroke. The completed vector overlay is cleared when a later committed frame refresh catches up, such as slide/load/pan/zoom/programmatic changes.
  - **Presentation modes:** full iPad sharing publishes the visible iPad canvas with external-display overlays; Present Mode publishes the centered 16:9 viewfinder crop so the TV fills correctly. The floating `•••` menu item reads as the action to switch to: `Present Mode` from full iPad sharing, `Mirror Mode` from Present Mode. Calculator remains visible on the external display in both modes; the radial palette is hidden in Present Mode.
  - **Important constraint:** do not reintroduce `drawHierarchy`, `layer.render(in:)`, or CADisplayLink snapshot mirroring for live PencilKit strokes. Those snapshot paths caused flicker/tearing and were replaced by the hybrid vector approach.

  - `IPHONEOS_DEPLOYMENT_TARGET = 26.0`, `MACOSX_DEPLOYMENT_TARGET = 26.0`

Current active direction:

- **Calculator follow-up polish is on hold.** Hardware testing confirms the darker calculator colors are easier to read and the calculator mirrors well to the TV in both Mirror and Present modes. Follow-up corrections separate the overscanned cache rect from the actual visible viewport rect and crop the overscanned `CGImage` to the current viewport before display, so the TV should match the iPad viewport and live ink should align. User retest pending for TV smoothness, any remaining edge artifacts, and the PDF touch-region jump.

Pen, Marker, and Eraser now drive PencilKit. Laser now works as a transient presentation tool with independent color/diameter/fade state. Text/equation now creates a persisted text object on tap, immediately selects the placeholder text for editing, keeps two-finger canvas pan/zoom available, hides the committed static copy while the live editor is active, and applies Bold/Italic/Underline style state to the active editor/object. Selection Object mode can select existing iPad text objects, show an outline, drag to move them, drag the lower-right handle to resize, double-tap to reopen editing, and use the Selection orbit Clone/Delete actions on the selected text object through a shared Canvas selection registry and explicit object commands. Object + Lasso/Box uses the drawn shape to select whole text/image objects; the dotted shape clears on lift and blue object highlights remain for Clone/Delete. Region + Lasso/Box keeps the dotted region shape as the selected region and intentionally does not highlight objects. Selection mode reserves one-finger/Pencil drags for lasso/box while keeping two-finger pan/zoom available; text/image pan only begins when the drag starts on an object or text resize handle. The lasso/box recognizer now uses the same low-latency touch sampler for Pencil and finger input, explicitly opts into UIKit Pencil touch routing, and disables the live-ink Pencil recognizer while Selection is active so the two recognizers do not compete. Reapplying the same Object/Lasso or Object/Box tool command no longer clears an existing object-region selection. The overlay renderer draws selected-object highlight boxes independently from the temporary dotted lasso/box path, so clearing the dotted path on lift does not also hide selected objects. Intended later Region behavior remains snapshot-style workflows first: copy/paste to another slide, sticker creation, and eventually a considered destructive cut/delete model. Widget remains UI-only until its canvas/object-layer implementation is built. **Geometry is now implemented** as its own object layer: selecting the Geometry tool arms a one-finger drag-to-create gesture (two-finger pan/zoom still works) that adds a `CanvasGeometryObject` (line, circle, right/isosceles triangle, rectangle, or regular polygon) styled by the palette's stroke color/width, fill color + `geometryFillOpacity`, `polygonSides`, and line `arrow` mode. In Select (Object) mode they tap-select, drag-move, resize, rotate, and Clone/Delete. **Selection handles** are drawn and hit-tested in screen space (via `CanvasGeometryObjectsView` and matching coordinator helpers) to avoid source/screen transform ambiguity: a bottom-right resize handle, a rotation knob on a stem above the top edge, and a **green pivot dot** (default center, draggable anywhere) about which rotation occurs. All handle interaction is rotation-aware (touches are un-rotated into the object's local space via `geometryLocalPoint`). Geometry participates in **region lasso/marquee object selection** alongside text/image (Select-tool drag on empty space, and region Clone/Delete). `rotation` (radians) and optional `pivotX/pivotY` persist in the sidecar and render everywhere via `CanvasGeometryRenderer` (which rotates about the mapped pivot; the PDF exporter flips its y-up context to reuse it). **Editing:** selecting a geometry object loads its properties into the palette and opens the geometry drawer so its controls edit the selected object live (via `CanvasObjectCommand.updateGeometry`) while the canvas stays in Select mode for continued manipulation. The compact palette gives Geometry a **quick shape mini-strip** (first tap shows shapes; second tap on the active Geometry tool opens the full drawer), and the line **arrow control is a Start/End pair of toggle buttons** (independent per-end on/off) instead of a slider.

## 8. Revisit later

This is the running list for issues that are acceptable for now but should be revisited before polish, beta, or whenever they become painful. Add to this list whenever we intentionally defer a known improvement.

- **Live external-display ink: pressure/taper matching closer to PencilKit.** Current live overlay uses a uniform SwiftUI path with a pressure-biased width multiplier, not PencilKit's true variable-width/tapered stroke model.
- **Live external-display ink: fade-out on the live overlay.** Add a short fade only if the pen-up swap remains noticeable after the committed PencilKit frame appears underneath.
- **Live external-display ink: tune the 150 ms clear delay lower if it feels sticky.** Current delay is intentionally conservative for AirPlay buffering.
- **Slide filmstrip ink thumbnails.** Filmstrip v1 shows PDF page thumbnails for PDF-backed slides and a placeholder for plain whiteboard slides. Later, render/cache `PKDrawing` thumbnails so ink-only slides preview their actual content.
- **Live external-display ink: possibly draw variable-width segments instead of one uniform SwiftUI path.** This would better match pressure and taper but is more work than the current acceptable implementation.
- **PDF import modes B/C need a page-object layer.** Future import modes "one slide with selected pages in an offset stack" and "place selected pages on current slide" should not be implemented as backgrounds. They require movable/resizable page objects with position, size, z-order, persistence, hit testing, external-display rendering, and selection/drag handles.
- **PDF export advanced modes.** Export v1 flattens selected slides into a new PDF and now includes basic current / PDF-backed / all / none selection presets. PDF-backed slide exports preserve the source PDF page as vector content, with PencilKit ink composited on top. Future polish should add the teacher-preferred guided-notes path for exporting selected pages from the same original source PDF as a packet, preserving original source order and offering richer presets like all PDF-backed slides from the same source PDF only.
- **Finger pinch / stretch / pan lag on iPad.** First mitigation implemented 2026-06-29 and user reported the iPad screen is noticeably more responsive. The likely hot path was confirmed: committed external-display frame publishing rasterized `PKDrawing.image(from:scale:2)` during scroll/zoom callbacks, even when no TV was connected. Current approach disables committed-frame publishing entirely when no external display is connected; when a TV is connected, it sends cheap source-rect updates during pan/zoom so the external display can transform the last committed frame live, while debounced/final raster frames refresh exact content. Committed TV frames are now rendered with a 2x source-rect overscan buffer to reduce black edges during live transforms. If lag persists, profile the remaining hot path: background/text-object frame updates, frequent viewport-state publication into SwiftUI, and PencilKit itself.
- **PDF page boundary cues.** Implemented and user-verified, but keep an eye on whether the subtle border/shadow is strong enough across projectors, TVs, iPad brightness levels, and exported PDFs.
- **PDF sharpness on iPad at high zoom.** Mac now uses vector PDF rendering for crisp high-zoom prep work. iPad may still rely on rasterized/composited paths in some places; revisit if high-zoom PDF text becomes a classroom problem.
- **Text object polish.** Text object v1 works on Mac and renders on iPad/external/PDF export. iPad can now create, edit, select, move, resize, duplicate, and delete text objects. Select-mode outside taps now attempt to close the active editor from the host canvas view; hardware retest is pending. Remaining polish includes richer keyboard behavior, text color controls for existing text, and switching the palette/tool state back to Select after Text-tool editing. Future LaTeX layout fix: resizing a text box should not scale equations. The box should act as a layout container, using its width for wrapping/available space while math remains rendered at the object font size; overflow should be handled by increasing height, clipping, or an overflow affordance rather than shrinking/stretching math.
- **Selection Region mode next steps.** Region mode now uses snapshot-style extraction for Copy, Clone, Sticker, and Send, and uses a clipped white fill patch for Delete. This is the right classroom-facing behavior for the first pass, but it is not yet a true layer/stroke destructive edit model. Future work: build the reusable sticker library UI/storage, support drag-and-drop stickers across slides, and revisit destructive pixel/stroke cutting only after the layer/object model is stronger.
